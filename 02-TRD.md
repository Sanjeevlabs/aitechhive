# AITechHive v2 — Technical Requirements Doc

Companion to `01-PRD.md`. This is the architecture.

---

## 1. Tech stack

| Layer | Choice | Why |
|---|---|---|
| Framework | **Next.js 15** (App Router) | Static + edge, plays well with Cloudflare Pages |
| Hosting | **Cloudflare Pages** | $0, unlimited bandwidth, edge runtime |
| Auth + DB | **Supabase** | Free 50K MAU. Google OAuth + magic-link built in. |
| LLM | **Provider-agnostic adapter** | One env var swaps 6 providers |
| Web search | Provider-native OR Brave fallback | Free tier, graceful degrade |
| Card archive | **`data/cards.json` in git** | Free, version-controlled, public |
| Cron | **GitHub Actions** | $0 (public repo = unlimited) |
| Analytics | **Plausible** | $0 free tier, no cookie banner |
| Image export | **`html-to-image`** | 30KB client-side, no server cost |

## 3. Environment variables

| Var | Required | Default | Purpose |
|---|---|---|---|
| `LLM_PROVIDER` | yes | `anthropic` | One of: anthropic, openai, gemini, groq, deepseek, openrouter |
| `LLM_API_KEY` | yes | — | API key for chosen provider |
| `LLM_MODEL` | no | provider default | Override default model |
| `BRAVE_API_KEY` | no | — | Enables web search fallback when provider lacks it |
| `NEXT_PUBLIC_SUPABASE_URL` | yes | — | Supabase project URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | yes | — | Supabase anon key |
| `SUPABASE_SERVICE_ROLE_KEY` | yes (server) | — | RLS bypass in cron |
| `NEXT_PUBLIC_PLAUSIBLE_DOMAIN` | no | `aitechhive.com` | Plausible domain |
| `NEXT_PUBLIC_APP_URL` | yes | `https://aitechhive.com` | For OG tags |

## 4. The LLM adapter — model freedom

`scripts/llm.mjs` exports:
```javascript
export async function generateCards({ systemPrompt, items, doWebSearch });
export function getProviderInfo();
```

Internally, a 6-provider config map. To switch:
```bash
LLM_PROVIDER=gemini
LLM_API_KEY=AI...
# That's it. No code changes.
```

5 providers (openai, gemini, groq, deepseek, openrouter) use the OpenAI SDK against their respective `baseURL`. Anthropic uses its own SDK. Common JSON output schema regardless.

## 5. Auth flow (Supabase)

```
User clicks "Continue with Google"
  → supabase.auth.signInWithOAuth({ provider: 'google' })
  → Google consent → redirect to /api/auth/callback?code=...
  → callback exchanges code for session, sets cookie
  → redirects to /
  → middleware.js refreshes session on every request
```

```
User clicks "Magic link"
  → supabase.auth.signInWithOtp({ email })
  → Email sent (Supabase's built-in templates)
  → User clicks link → callback → session set
```

## 6. Performance budgets

| Metric | Budget |
|---|---|
| FCP (mobile, 3G) | ≤ 1.8s |
| LCP | ≤ 2.5s |
| TBT | ≤ 200ms |
| CLS | ≤ 0.05 |
| JS initial | ≤ 180KB gz |
| Lighthouse mobile | ≥ 90 |

How:
- Static generation for `/`, `/privacy`, `/terms`
- Cards JSON fetched at build, revalidated every 4min
- Framer Motion lazy-imported
- `html-to-image` only loaded on first share intent
- `next/font` for zero layout shift

## 7. Security

| Concern | Mitigation |
|---|---|
| XSS | All card text via React (escaped). No `dangerouslySetInnerHTML`. |
| CSRF | Supabase JWT cookie validated via middleware. |
| Rate limiting | Cloudflare per-IP 60 req/min for `/api/*`. Supabase RLS. |
| Secret exposure | All `LLM_API_KEY`, service role keys are server-only env vars. |
| Email harvesting | Emails never in client bundle. Supabase auth handles. |
| RLS bypass | Service role used only in cron. Never in client. |

## 8. Observability

We log:
- Cron run timing (GitHub Actions logs)
- LLM cost per call (stdout from `enrich.mjs`)
- API errors (Cloudflare Pages logs)
- User events (Plausible)

We don't log:
- Full request bodies
- User emails outside Supabase
- IPs beyond rate limiting
